## Problem Statement

한 회사에 여러 팀의 AI 에이전트(고객지원 봇, 데이터 분석 에이전트, 개발자 어시스턴트)가 있고, 각자 다른 권한으로 사내 도구(티켓, 문서, 운영 데이터)에 접근해야 한다. 에이전트가 MCP 서버에 직접 연결하는 현재 구조에서는 "support-agent는 ops 데이터에 접근 불가" 같은 권한 매트릭스를 tool call 단위로 강제할 방법이 없고, 누가 무엇에 접근 시도했는지 감사할 방법도 없다. 이 프로젝트는 그 한가운데에 Gateway를 놓아 라우팅·인증·정책·관측을 단일 진입점에서 해결한다.

검증 환경은 외부 의존성 없는 자체 시뮬레이션: 직접 구현하는 백엔드 MCP 서버 3종 + 가상 에이전트 클라이언트 3종 + 아래 권한 매트릭스.

**백엔드 서버와 tool 목록:**

| 서버 | tools | 성격 |
|---|---|---|
| ticket-server | `create_ticket`, `search_tickets`, `update_status` | SQLite 기반. 쓰기 있음 — 정책 테스트용 |
| docs-server | `search_docs`, `read_doc` | 마크다운 저장소 + BM25 검색. 읽기 전용 대표 |
| ops-server | `get_metrics`, `query_logs` | 가짜 운영 데이터. "민감 데이터" 역할 |

**권한 매트릭스 (3×3 — 프로젝트 전체의 뼈대):**

| 에이전트 | ticket | docs | ops |
|---|---|---|---|
| support-agent | 읽기+쓰기 | 읽기 | ❌ 차단 |
| analyst-agent | 읽기 | 읽기 | 읽기 |
| dev-agent | 읽기+쓰기 | 읽기 | 읽기+쓰기 |

## Demand Evidence

**현재 0. 솔직하게 기록한다.** 사용자는 특정 회사/팀을 염두에 두고 있으나 아직 해당 회사와 이 문제로 대화하지 않았다 (세션 중 직접 확인). 카테고리 수준 검증은 존재한다:

- MCP gateway 카테고리에 다수의 상용/OSS 플레이어 존재 (시장이 문제를 인정한다는 간접 증거)
- CIS가 엔터프라이즈 AI 에이전트 거버넌스 가이드(CIS Controls v8.1 적용)를 발간 — 거버넌스 수요의 제도적 신호
- 단, 이것은 "이 제품"에 대한 수요 증거가 아니라 "이 카테고리"에 대한 증거다

## Status Quo

- 에이전트는 MCP 서버에 직접 연결하고, 권한은 서버별 크리덴셜 수준에서만 통제됨. tool call 단위 정책·인자 레벨 제한·감사 로그는 없음
- 시장 측 status quo: IBM ContextForge(OSS, 3k+ stars, 플러그인 아키텍처), Lasso Security(보안 특화 OSS), MintMCP, TrueFoundry, agentic-community/mcp-gateway-registry 등이 이미 이 문제를 풀고 있음 — 이 카테고리는 레드오션

## Target User & Narrowest Wedge

- **가설 타깃 (미검증)**: 에이전트 도입을 검토하다가 "도구 접근을 통제할 방법이 없다"는 이유로 보류시키는 보안/플랫폼 담당자
- **잠재 wedge (이번 범위에선 채택 안 함)**: 범용 gateway가 아닌 거버넌스 증거 계층 — append-only audit log + "지난 24시간 누가 민감 tool에 접근 시도했나" 리포트. Approach C로 제안되었으나 사용자가 B(관측성 깊이 우선)를 선택. wedge 탐색은 The Assignment의 첫 대화 결과에 따라 다음 세션에서 재검토

## Constraints

- 솔로 개발, 목표 4주 (Week 4는 선택)
- 외부 의존성 없는 완전 통제 가능한 시뮬레이션 환경 — **Week 1~3 한정**. Week 4의 LangGraph 에이전트만 실 LLM API를 사용 (유일한 예외)
- 스택: FastAPI 기반 Gateway, Streamable HTTP transport, SQLite(ticket), Prometheus + Grafana(Docker), LangGraph(Week 4)
- 프로젝트 CLAUDE.md의 Karpathy 4원칙 가드레일 적용

## Premises (세션에서 합의됨)

1. **[산출물 성격 — 사용자 이의 후 수정·확정]** 이 4주 산출물은 '고객이 쓰는 제품'이 아니라 '시뮬레이션 환경 위의 데모 시스템'이다. 사용자는 처음 이 전제에 이의를 제기했으나("특정 회사/팀이 있다"), 구체성 검증에서 "아직 대화 전"임을 확인하고 수용. 실서비스 경로는 코드가 아니라 **첫 대상 회사와의 대화**라는 별도 단계다.
2. **[경쟁/wedge]** 범용 MCP Gateway로는 IBM ContextForge와 정면승부가 안 된다. 스타트업으로 성립하려면 좁은 wedge가 필요하다. (동의됨 — 단 wedge 채택은 보류, 위 Target User 참조)
3. **[뼈대]** 권한 매트릭스(3 에이전트 × 3 서버) + tool call 단위 정책 enforcement가 프로젝트의 핵심 가치다. "Gateway가 없으면 이 매트릭스를 강제할 방법이 없다"가 곧 존재 이유다.
4. **[데모]** "정책 거부 → 에이전트가 거부를 받고 우회 계획" 장면과 audit log가 가장 설득력 있는 산출물이다.

## Approaches Considered

### Approach A: Policy Demo Cut — 2주 최소판
Gateway 코어 + JWT/YAML 정책 엔진 + 정책 거부 데모까지만. 관측 스택·LangGraph 제외. Effort S-M, Risk Low. 빠르지만 관측·거버넌스 시그널이 빠져 토이로 보일 위험.

### Approach B: 원안 풀 플랜 — 4주 완주 ★ 선택됨
- **Week 1**: FastAPI Gateway 코어 — Streamable HTTP transport, tools/list aggregation(`ticket__`/`docs__`/`ops__` prefix 네임스페이싱), prefix 기반 라우팅, 백엔드 세션 유지(백엔드당 MCP 세션 1개를 열어 재사용, 커넥션 풀 아님; 동시 호출은 세션을 공유하고, 끊김 시 재연결은 lock 뒤에서 직렬화). **tools/list는 정책으로 필터링하지 않는다** — 모든 에이전트가 전체 aggregated 목록을 보고, 거부는 호출 시점에 발생. 의도적 선택: support-agent가 ops tool의 존재를 알아야 Week 4의 "호출 시도 → 거부" 장면이 자연스럽게 발생한다
- **Week 2**: JWT 인증 + YAML 정책 엔진 + audit log 기록. 세부 명세는 아래 "구현 명세 결정사항" 참조
- **Week 3**: Observability — 전 호출 trace ID, OpenTelemetry 계측, Prometheus + Grafana 대시보드(클라이언트별/tool별 호출량, latency p50/p99, 정책 거부 카운트). 우선순위 순서: 정책 거부 카운트 메트릭 > 대시보드 > rate limiting(클라이언트별 token bucket) > circuit breaker > OPA/Rego 교체 — 뒤 3개는 stretch goal
- **Week 4 (선택)**: LangGraph로 support-agent 구현, Gateway 경유 동작 + "ops tool 호출 → 정책 거부 → 우회 계획" 데모 + audit admin 페이지("지난 24시간 누가 민감 tool에 접근 시도했나")
- Effort L, Risk Med-High (Week 3가 솔로 프로젝트의 통상 이탈 지점)

**구현 명세 결정사항 (Week 2 핵심):**
- **JWT**: 데모용 정적 사전발급 — 셋업 스크립트가 에이전트 3개분 토큰을 생성. claim은 `{agent_id, exp}`, 만료는 데모 편의상 30일. 발급 서버는 만들지 않는다. 서명은 HS256, secret은 env var(`GATEWAY_JWT_SECRET`), 전송은 `Authorization: Bearer` 헤더
- **YAML 정책**: **default-deny**. 매트릭스에 없는 (에이전트, tool) 조합은 전부 거부. 스키마 예시:
  ```yaml
  support-agent:
    ticket: [create_ticket, search_tickets, update_status]
    docs: [search_docs, read_doc]
    # ops 미기재 = 전부 거부
  ```
- **인자 레벨 정책**: analyst-agent의 `query_logs` 시간 범위 최대 24시간. ops-server의 tool 시그니처를 `query_logs(query: str, start: ISO8601, end: ISO8601)`로 정의하고, 검사는 `end - start <= 24h`. 위반 시 클램핑이 아니라 **거부** (정책 위반을 가시화하는 게 목적이므로)
- **거부 응답 형식**: MCP tool result `isError: true` + 구조화 payload `{"code": "POLICY_DENIED", "rule": "<agent>:<server>:<tool>", "agent": "<agent_id>"}` — `rule`은 YAML 매핑에서 그대로 도출 가능한 형식. 에이전트가 파싱해 우회 계획을 세울 수 있는 형식 (Week 4 데모의 전제). rate limit 초과는 동일 형식에 `code: "RATE_LIMITED"`
- **audit log**: append-only JSONL — 모든 tool call의 (timestamp, agent, tool, 인자 요약, 허용/거부, trace ID). 인자 요약은 전체 인자 JSON을 256자에서 절단한 것. **기록 자체는 Week 2 정책 엔진에 포함** (비용 미미), admin 페이지만 Week 4
- **circuit breaker (stretch)**: 연속 실패 N회 시 open → 30초 후 half-open으로 1회 probe → 성공 시 close. open 동안 해당 백엔드 tool은 tools/list에서 제외

### Approach C: Governance Wedge — 4주, 차별화 재배치
코어·정책·데모는 원안대로, Week 3 관측을 1장 대시보드로 축소하고 audit log + 거버넌스 admin + 거부-우회 데모에 선투자. AI가 권고했으나 사용자가 기각.

## Recommended Approach

**Approach B (원안 풀 플랜)** — 사용자 선택. 근거: 관측성 스택(OTel, Prometheus, Grafana)이 사용자의 커리어 축(관측성 스택 딥다이브, LangGraph 관측 시스템 경험)과 직결되며, 면접 시그널로서 엔지니어링 폭을 최대로 보여준다. AI 권고는 C였으나, B 선택은 "이력서 키워드 커버"라는 사용자만 아는 맥락에서 일관된 판단이다.

**리스크 완화 조건**: Week 3에 진입하기 전 Week 1-2 산출물(코어 + 정책 + 매트릭스 9칸 테스트)이 완결되어 있어야 한다. Week 3에서 시간이 모자라면 Grafana 패널 수를 줄이지, 정책 거부 카운트 메트릭은 절대 빼지 않는다(전제 4).

## Success Criteria

- 권한 매트릭스 9칸(3 에이전트 × 3 서버) 전부가 통합 테스트로 검증됨 — 허용 칸은 성공, 금지 칸은 정책 거부 응답
- 인자 레벨 정책 1개(query_logs 시간 범위) 테스트 통과
- 스크립트 클라이언트(LLM 불필요)가 support-agent 토큰으로 ticket 생성 → docs 검색 → ops 호출 시도 → `POLICY_DENIED` 응답 수신까지 E2E 동작 — **Week 3 종료 시점 기준**
- (Week 4 진행 시) LangGraph support-agent가 위 시나리오를 실제 LLM 추론으로 수행하고, 거부 응답을 받아 우회 계획을 출력
- audit log JSONL에 허용/거부가 trace ID와 함께 기록됨
- Grafana에서 클라이언트별 호출량 / latency p50·p99 / 정책 거부 카운트 확인 가능
- (stretch 달성 시) 백엔드 1개 강제 다운 시 circuit breaker가 해당 백엔드 tool을 aggregation에서 제외
- `docker compose up` 한 번으로 전체 데모 환경 기동

## Distribution Plan

- 데모 단계: GitHub public repo + `docker compose up` 원커맨드 기동 + README에 데모 GIF(정책 거부 장면)

## Dependencies

- MCP Python SDK (Streamable HTTP transport 지원 버전)
- Docker (Prometheus + Grafana, Week 3)
- LangGraph + LLM API 키 (Week 4)
- 선행 작업 없음 — 빈 레포에서 시작

## The Assignment

**코드보다 먼저, 늦어도 Week 1과 병행해서: 염두에 둔 그 회사의 담당자와 첫 대화를 잡아라.** 질문은 하나면 된다 — "에이전트가 사내 도구에 접근하는 걸 지금 어떻게 통제하고 있어요?" 팔러 가는 게 아니라 들으러 가는 것이다. 이 답이 (1) Week 4 데모 시나리오를 실제 상황으로 바꿔주고, (2) 전제 1을 수정할 유일한 증거이며, (3) 면접에서 "사용자 검증도 했다"는 한 줄이 된다. 4주 뒤 데모가 완성됐을 때 보여줄 상대가 이미 확보되어 있는 것과 아닌 것의 차이는 크다.

## What I noticed about how you think

- 전제 1(이건 데모다)에 이의를 제기했고, "수요 증거가 뭐냐"는 압박에 "아직 대화 전이다"를 선택했다. 반박하다가 증거 앞에서 정직하게 물러서는 것 — 이게 검증 가능한 사고방식이고, 대부분은 여기서 포장한다.
- AI 권고(C)를 알고도 B를 골랐다. 근거 없는 고집이 아니라 "관측성 스택이 내 커리어 축"이라는, 세션 밖의 본인 맥락에 기반한 일관된 선택이었다. 권고를 기각할 줄 아는 건 신호다.
- 가져온 플랜 자체가 이미 "권한 매트릭스 하나가 프로젝트 전체의 뼈대"라는 문장을 담고 있었다. 기능 목록이 아니라 존재 이유부터 정의한 플랜은 드물다.
- 다만 "특정 회사/팀이 있다"에서 멈췄고 이름·역할·들은 말은 없었다. 다음 세션에선 그 빈칸이 채워져 있길 바란다 — The Assignment가 그걸 위한 것이다.
